# 版本命名規則

## 格式
```
v{major}.{minor}.{patch}-{category}
```

## Category 後綴說明

### `-ml` (Machine Learning)
用於機器學習模型、訓練、預測相關的更新
- 例：模型訓練改進、特徵工程、回測優化

### `-ui` (User Interface)
用於 Web UI、視覺化、使用者體驗相關的更新
- 例：頁面設計、圖表優化、互動改進

### `-verified` (Verification & Testing)
用於測試系統、驗證機制、品質保證相關的更新
- 例：自動化測試腳本、CI/CD 整合、驗證工具

### `-data` (Data Pipeline)
用於資料處理、ETL、資料源整合相關的更新
- 例：新增資料源、資料清理邏輯、指標計算

## 版本號遞增規則

### Major (主版本號)
- 重大架構變更
- 不向後相容的改動
- 全新功能模組

### Minor (次版本號)
- 新功能新增
- 向後相容的改進
- 跨模組整合

### Patch (修訂號)
- Bug 修復
- 小幅優化
- 文件更新

## 範例

- `v2.0.0-ml` - 首次引入機器學習模型
- `v2.1.0-ml` - 新增模型自動化訓練
- `v2.4.0-ui` - UI 大幅改版
- `v2.5.0-verified` - 整合驗證系統（跨 ML + UI 的品質保證）

## 特殊情況

當一個版本**整合多個模組**時（例如同時涉及 ML、UI、驗證），使用最能代表該版本**核心價值**的 category。

例如：
- 若主要是「建立測試框架以確保 ML + UI 穩定性」→ 使用 `-verified`
- 若主要是「UI 改進並順便修了 ML bug」→ 使用 `-ui`
